# Bes3 后台管理系统完整规划方案 (Admin Console Blueprint)

## 1. 设计结论

Bes3 需要一套独立、完整、强运营属性的后台管理系统。它不应被设计成传统 CMS，而应定位为 **Bes3 Evidence Operations Console**：用于管理商品 SKU、视频证据、标签体系、AI 抽取、价格价值、pSEO/GEO、联盟链接健康、风控合规、系统任务和运行配置。

对外网站负责呈现“消费者可理解的结论”，后台负责保证这些结论 **可追溯、可重跑、可修正、可发布、可审计**。

## 2. 借鉴 autobb 的后台经验

`/Users/jason/Documents/Kiro/autobb/` 的后台系统给 Bes3 提供了几个关键参考：

| autobb 模式 | 代表位置 | Bes3 可借鉴方式 |
| --- | --- | --- |
| 受保护应用区 | `src/app/(app)/layout.tsx`, `src/components/layout/AppLayout.tsx` | Bes3 后台继续使用 `/admin` 受保护区，但导航应从“内容生成”扩展到“证据运营”。 |
| 分域导航 | Dashboard, Products, Offers, Campaigns, Data Management, Risk Alerts, Settings | Bes3 应按业务闭环分区：证据、商品、标签、价格、pSEO、风控、系统。 |
| 仪表盘优先 | `DashboardClientPage.tsx` | 首页不放大营销叙事，而显示任务健康、证据覆盖、断链风险、价格刷新、pSEO 状态。 |
| 可操作对象列表 | `products/page.tsx`, Offers 管理 | Bes3 商品、视频、证据、标签、页面、链接都应有列表、筛选、批量操作。 |

| 运行健康监控 | `admin/proxy-health/page.tsx`, performance APIs | Bes3 需要代理池、YouTube 抓取、AI 额度、Google indexing、价格源、联盟 API 的健康面板。 |

核心原则：**借鉴 autobb 的系统化后台骨架，不照搬广告投放业务域。Bes3 的后台主语是 Evidence Graph，而不是 Campaign。**

### 2.1 生产登录机制借鉴

autobb 的登录机制比 Bes3 当前实现更接近生产要求。Bes3 现状已有 `HttpOnly auth_token`、`SameSite=Lax`、生产环境 `Secure` cookie、`/api/auth/login`、`/api/auth/me`、`/api/auth/logout` 和 middleware 保护 `/admin`、`/api/admin`，但仍缺少失败登录治理、服务端会话表、强制改密、登录审计和多实例限流。

Bes3 后台登录必须按以下生产基线补齐：

| 能力 | autobb 做法 | Bes3 落地要求 |
| --- | --- | --- |
| 输入校验 | login API 使用 zod 校验用户名/邮箱和密码。 | `/api/auth/login` 必须使用 schema 校验，并统一返回错误，避免账号枚举。 |
| IP 与设备识别 | 读取 `x-forwarded-for`、`x-real-ip` 和 `User-Agent`。 | 所有登录、登出、会话刷新、敏感操作都记录代理后的真实 IP、UA、request id。 |
| 登录限流 | 同时按 IP 和 username/email 做 rate limit。 | 单实例可先内存限流，生产多实例必须接 Redis 或等价共享存储。 |
| 失败登录治理 | 记录 `login_attempts`，多次失败后禁用账号。 | 连续失败达到阈值后锁定账号，必须由 Admin 解锁并写入审计。 |
| 账号状态 | 登录前检查 `is_active`。 | 所有 admin API 均二次检查用户仍 active，不能只信 JWT。 |
| 强制改密 | JWT 内包含 `mustChangePassword`，middleware 限制可访问路径。 | bootstrap 默认管理员必须 `must_change_password=true`，首次登录只能进入改密流程。 |
| 密码强度 | 改密页要求长度、大小写、数字、特殊字符。 | 密码策略至少 12 位，建议 16 位；禁止默认占位密码进入生产。 |
| 会话记录 | `user_sessions` 记录 token/session、IP、UA、过期、撤销。 | JWT cookie 之外必须有服务端 session row，登出和封禁可立即撤销。 |
| 并发会话 | 限制最多 3 个活跃会话。 | 后台用户默认最多 3 个活跃会话，Admin 可在 Users & Access 中踢下线。 |
| 可疑登录 | 识别短时间多 IP、新设备、账号共享。 | 生成 `admin_security_events`，高风险时要求改密或人工确认。 |
| Cookie | `HttpOnly`、生产 `Secure`、`SameSite=Lax`、7 天。 | 保持当前 cookie 基线，生产必须依赖 HTTPS，反向代理需正确传递 proto/header。 |

Bes3 不应照搬 autobb 的套餐过期逻辑。autobb 的 `package_type/package_expires_at` 属于 SaaS 授权场景；Bes3 后台生产基线应改成 **角色、账号状态、会话状态、敏感操作二次确认**。

### 2.2 生产服务启动机制借鉴

autobb 的生产启动经验集中在三点：启动前强预检、数据库/管理员初始化前置、用 supervisor 托管 web/worker/网关等长进程。Bes3 现状已经具备 `Dockerfile`、`docker-compose.yml`、`supervisord.conf`、`scripts/check-runtime-env.js`、`scripts/db-init.ts`、`scripts/migrate.ts`、`scripts/worker-standalone.ts` 和 `/api/health`，但还需要把启动顺序、失败策略和上线验收写成硬规则。

Bes3 生产启动链路应定义为：

```text
container start
  -> load .env.production
  -> run ops:check-env
  -> ensure data/media directories and permissions
  -> run db:init
  -> run db:migrate
  -> run admin:ensure with forced password change
  -> start Next.js web process
  -> start pipeline worker process
  -> expose /api/health readiness
  -> supervisor restarts failed long-running processes
```

启动机制要求：

| 能力 | 当前 Bes3 状态 | 生产要求 |
| --- | --- | --- |
| 环境变量预检 | 已有 `ops:check-env`，覆盖 APP URL、JWT、管理员密码、数据库、媒体、AI、PartnerBoost、worker。 | Docker 启动必须先执行，失败立即退出；禁止默认 `JWT_SECRET` 和默认管理员密码进入生产。 |
| 数据初始化 | 已有 `db:init` 和 `bootstrapApplication`。 | 入口脚本必须在 web 启动前初始化 schema、管理员账号和必要 settings。 |
| 增量迁移 | 已有 `db:migrate`，支持 SQLite/PostgreSQL。 | 生产启动应在 `db:init` 后运行迁移，迁移失败不得继续启动。 |
| 进程托管 | 已有 `bes3-web`、`bes3-worker` 两个 supervisor program。 | web 与 worker 分开托管，日志进入 stdout 或明确挂载目录，失败自动重启。 |
| 健康检查 | 已有 `/api/health` 和 `ops:health-check`。 | health 必须返回数据库连接、worker 配置、媒体驱动；未来增加外部依赖 degraded 状态。 |
| 持久化目录 | Docker 暴露 `/app/data`、`/app/storage/media`。 | SQLite、媒体、备份必须挂载；生产推荐 PostgreSQL + S3/R2，容器本地只存临时文件。 |
| Worker | 已有独立 pipeline worker。 | worker 必须可禁用、可限并发、可观察；后台 Pipeline Queue 显示 worker 心跳和失败。 |
| 日志 | supervisor 当前写本地 log 文件。 | 容器化部署建议 stdout/stderr，保留 request id，方便接入云日志。 |
| 优雅停止 | supervisor 已 `stopasgroup/killasgroup`。 | web/worker 需响应 SIGTERM，worker 停止时不丢正在执行的任务。 |

Bes3 的对外网站必须保持公开可抓取；后台登录保护只能作用于 `/admin/*` 和 `/api/admin/*`。`/products`、`/reviews`、`/guides`、`/deals`、`/llms.txt`、`/sitemap.xml`、`/robots.txt`、`/api/open/*` 等公开面不能被后台 session 影响。

## 3. Bes3 后台产品定位

### 3.1 后台使命

后台要回答 7 个运营问题：

1. 现在有哪些商品可以公开推荐？
2. 每个推荐结论背后的证据是否足够、准确、可追溯？
3. 视频、商品、ASIN、联盟链接是否对齐？
4. 哪些标签和场景正在增长，哪些应该降级或暂停？
5. 当前价格是否处在值得购买的窗口？
6. 哪些 pSEO 页面可以 index，哪些必须保持 researching/noindex？
7. 系统依赖、任务队列、外部 API、合规声明是否健康？

### 3.2 使用角色

| 角色 | 主要目标 | 权限边界 |
| --- | --- | --- |
| Admin | 系统配置、用户、密钥、任务控制、发布策略 | 全权限，包括敏感配置和删除/回滚。 |
| Evidence Ops | 视频导入、证据审核、SKU 对齐、反馈处理 | 可改证据与匹配状态，不可改密钥。 |
| Content/SEO Editor | pSEO 页面、FAQ、标题、索引、内容质量 | 可发布/暂停页面，不可改核心评分公式。 |
| Commerce Ops | 联盟链接、价格、库存、商家、告警 | 可改链接健康、价格源、商家状态。 |
| Viewer | 只读巡检、业务复盘 | 不可触发写操作。 |

## 5. 核心模块设计

### 5.1 Command Center

首页应该是运维控制面板，不是欢迎页。

关键区域：

| 区域 | 内容 |
| --- | --- |
| Evidence Readiness | 有证据商品数、无证据商品数、低置信证据、争议商品、冻结商品。 |
| Commerce Readiness | 可购买商品、断链、缺货、未验证链接、价格基线缺失。 |
| pSEO Readiness | live 页面、researching/noindex 页面、受影响页面、最近 indexing 事件。 |
| Pipeline Health | queued/running/failed/cancelled，平均耗时，失败最多阶段。 |
| High-Value Actions | 导入商品、导入视频、运行 rescan、刷新价格、运行 link inspector、推送 pSEO。 |
| Alerts | 最高优先级风险卡片，点击进入 Risk Center。 |

交互要求：

- 顶部保留时间范围选择：24h / 7d / 30d / all。
- 所有指标都必须可点击进入对应列表。
- 所有批量动作必须有确认弹窗，并显示影响范围。
- 刷新类按钮要显示 loading、成功、失败 toast。

### 5.2 Products & SKUs

这是 Bes3 的核心对象管理区。

页面结构：

1. 商品列表
   - 筛选：category、source platform、affiliate status、evidence status、price status、published status。
   - 排序：updated_at、consensus score、value score、evidence count、link health、price freshness。
   - 批量操作：queue pipeline、refresh price、inspect links、export selected、mark researching。

2. 商品工作台
   - 基础信息：brand、model、category、slug、ASIN、source links。
   - SKU 对齐：关联 affiliate product、matched videos、entity match confidence。
   - Evidence summary：共识分、证据数量、来源频道、争议/冻结状态。
   - Price-value：current price、hist low、90d avg、entry status、alert count。
   - Public surface：product page、scenario pages、deal pages、schema status。
   - Actions：rerun full pipeline、rerun entity resolution、refresh price、rescan evidence、publish/noindex。

3. SKU Review Queue
   - 展示 entity match confidence 低于阈值、ASIN 冲突、视频描述链接冲突、品牌型号模糊的项目。
   - 支持人工 accept/reject/merge/split。

### 5.3 Evidence Graph

后台必须让操作者看到“推荐依据”，而不是只看到商品。

子页面：

| 页面 | 功能 |
| --- | --- |
| Review Videos | 视频库、YouTube ID、频道、标题、字幕、描述、处理状态。 |
| Evidence Reports | `analysis_reports` 列表，按商品、标签、rating、confidence、source 过滤。 |
| Extraction Review | AI 抽取结果审核，显示 quote、context、timestamp、原字幕片段。 |
| Feedback Queue | 用户/内部反馈，包括 inaccurate、wrong product、bad quote、useful。 |
| Creator Attribution | 检查 timestamp link、channel link、Review by 文案是否完整。 |

证据详情页应显示：

- 原始 transcript 片段。
- AI 输出 JSON。
- hard validation 结果。
- allowed tags 命中情况。
- evidence quote 是否可在 transcript 中找到。
- 该证据对共识分的权重贡献。
- 操作：approve、downgrade confidence、mark advertorial、detach product、queue re-extraction。

### 5.4 Taxonomy Lab

这个模块负责标签体系自进化。

核心列表：

| 对象 | 数据来源 | 操作 |
| --- | --- | --- |
| Intent Sources | Amazon autosuggest、Google Keyword Planner、Reddit、site search | normalize、dedupe、promote to pending。 |
| Pending Tags | 站内搜索/外部数据生成的新候选标签 | approve、merge、reject、promote。 |
| Taxonomy Tags | 已生效标签 | edit canonical name、keywords、core painpoint、pause/resume。 |
| Rescan Queue | 新标签触发的视频回扫任务 | mark processing、retry、cancel、export jobs。 |
| pSEO Signals | GA4/search signal | boost、downgrade、pause low-priority tags。 |

关键 UX：

- 标签合并必须显示受影响页面和证据数量。
- Promote 操作必须生成 rescan job。
- Pause 标签必须显示会 noindex 或降权的页面列表。

### 5.5 Price & Value

这个模块承接 planv2 的价格价值算法。

页面：

| 页面 | 功能 |
| --- | --- |
| Price Snapshot Dashboard | 当前价格、历史低价、90d 均价、entry status 分布。 |
| Product Price Detail | 单商品价格历史、value score 趋势、来源记录。 |
| Deal Page Manager | `/deals/best-value-{category}-under-{price}` 页面状态。 |
| Price Alerts | 用户价格提醒、触发条件、notification queue、发送状态。 |
| Price Source Health | PartnerBoost/merchant scrape/API 当前可用性。 |

必须支持：

- dry-run refresh：预览不写库。
- real refresh：写入 `price_value_snapshots` 并更新 products。
- alert evaluation：可 dry-run、queue notification、mark notified。
- dispatch：Webhook/email 发送，失败可重试。

### 5.6 pSEO/GEO Ops

现有 `/admin/seo-ops` 应扩展成 pSEO/GEO 总控。

页面区域：

| 区域 | 内容 |
| --- | --- |
| URL Matrix | Category pages、single-tag pages、multi-constraint pages、deal pages。 |
| Index State | live/researching/noindex、canonical、schema、FAQ、last crawled。 |
| Content Requirements | BLUF、HTML table、Evidence Stream、FAQPage、Product schema 是否完整。 |
| GA4 Signal Intake | impression/click/CTR，导入记录，影响到 taxonomy 的变更。 |
| LLM Surface | robots、llms.txt、open data API、feed、media sitemap、security.txt。 |
| Distribution | Google indexing、syndication、Reddit kit、screenshot capture。 |

操作：

- regenerate route manifest。
- import pSEO signals。
- apply signals to taxonomy。
- run trust surface audit。
- run link inspector。
- push indexing。
- export Reddit reply kit。

### 5.7 Pipeline Queue

借鉴 autobb `admin/queue`，但 Bes3 的任务类型应围绕证据和内容：

| Task Type | 说明 |
| --- | --- |
| affiliate-product-sync | PartnerBoost/DTC 商品同步。 |
| full-product-pipeline | 商品抓取、媒体、关键词、文章、SEO。 |
| video-transcript-import | YouTube metadata/transcript 导入。 |
| entity-resolution | 视频和商品 SKU 对齐。 |
| evidence-extraction | AI 从 transcript 抽取证据。 |
| taxonomy-rescan | 新标签回扫旧视频。 |
| price-value-refresh | 刷新价格快照。 |
| price-alert-evaluation | 评估价格提醒。 |
| link-inspection | 检查联盟链接。 |
| pseo-push | pSEO indexing/syndication。 |

队列页必须支持：

- 按状态、类型、worker、时间过滤。
- 查看 job-level trace。
- cancel/retry/requeue。
- 显示错误信息和 payload。
- 并发设置和 per-type limit。
- zombie task 检测。

### 5.8 Risk Center

Bes3 后台最重要的质量模块之一。

风险类型：

| 风险 | 触发条件 | 处理动作 |
| --- | --- | --- |
| Entity mismatch | ASIN 冲突、brand/model 置信度低、视频多商品混杂 | 进入 SKU review。 |
| Weak evidence | 证据少于阈值、quote 不在 transcript、无 timestamp | 降级 confidence 或回抽。 |
| Shill risk | advertorial 高占比、低权重频道异常好评 | freeze recommendation。 |
| Controversy | 标准差高、正反结论冲突 | 标记 highly controversial。 |
| Broken commerce | 断链、缺货、affiliate status broken | 降级 CTA、推荐替代品。 |
| Price stale | 价格超过刷新周期 | queue refresh。 |
| SEO invalid | live 页面缺 schema、FAQ、BLUF 或 evidence table | 进入 remediation queue。 |
| Compliance gap | 缺 FTC disclosure、cookie control、creator attribution | 阻止 publish。 |

Risk Center 应该是所有“不能公开推荐”的原因入口。

### 5.9 Prompt Lab

现有 Prompt 管理应升级为 AI 工程面板。

功能：

- Prompt versioning：taxonomy refinement、long video extraction、shorts extraction、entity extraction。
- Schema registry：Zod schema、required fields、validation rules。
- Failed output samples：保存验证失败的 AI 输出。
- Retry prompt preview：展示系统生成的 retry prompt。
- Regression set：固定 transcript 样本，比较 prompt 改版前后输出。
- Publish guard：prompt 未通过回归测试不可设为 active。

### 5.10 Data Management

借鉴 autobb Data Management，但对象换成 Bes3 数据域。

导出对象：

- Products/SKUs。
- Review videos。
- Evidence reports。
- Taxonomy tags。
- Pending tags。
- Price snapshots。
- SEO pages。
- Link inspector results。
- Settings，敏感字段默认脱敏。

导入对象：

- Keyword Planner CSV。
- Manual intent source CSV/JSON。
- Shorts evidence JSON。
- pSEO signal CSV/JSON。
- Manual product seed JSON。

安全要求：

- 敏感配置默认不导出。
- 导入必须先 dry-run，显示会新增/更新/跳过/冲突的行数。
- 所有导入写操作进入 audit log。

## 6. API 设计建议

Bes3 现有 `/api/admin/*` 可以扩展为以下结构：

| API | 方法 | 说明 |

## 7. 数据模型补充

现有 schema 已覆盖大部分 planv2 表。后台完整化还需要补充或正式化这些表：

| 表 | 用途 |
| --- | --- |
| `admin_audit_logs` | 记录后台所有写操作、操作者、payload diff、IP、user agent。 |
| `admin_risk_alerts` | 聚合实体、证据、链接、SEO、价格、合规风险。 |
| `admin_saved_views` | 保存列表筛选视图，例如 “low confidence evidence”。 |
| `admin_import_runs` | 记录 CSV/JSON 导入 dry-run 和 real-run 结果。 |

生产登录与访问控制需要正式化以下表：

| 表 | 用途 |
| --- | --- |
| `admin_users` 或扩展现有 `users` | 后台账号、角色、状态、强制改密、最后登录时间。 |
| `admin_login_attempts` | 按账号和 IP 记录成功/失败、失败原因、request id。 |

会话表是生产必需项。仅依赖 7 天 JWT cookie 会导致账号禁用、改密和踢下线不能立即生效；Bes3 后台必须在每次读取 session 时检查服务端 session row 和用户状态。

## 8. UI/UX 设计规范

基于 `ui-ux-pro-max` 原则，Bes3 后台应采用安静、高密度、可扫描的运营工具风格：

- 不使用营销式 hero，不做大面积装饰插画。
- 卡片用于指标、队列项、告警项，不把页面区块都做成浮动卡片。
- 表格必须支持筛选、排序、分页、批量选择、空状态。
- 危险操作必须二次确认，并显示影响对象数量。
- 所有异步动作必须有 loading、toast、错误详情。
- 图标使用 lucide-react，不使用 emoji。
- 页面标题规模收敛，后台内部不要使用 public site 的大标题风格。
- 保持 44px 以上触控目标，所有 icon-only button 必须有 aria-label 或 tooltip。
- 数据列里的数字使用 tabular/monospace 风格，避免刷新时宽度跳动。
- 移动端只保证可查看和轻操作，批量审核/复杂配置以桌面为主。

推荐布局：

| 层级 | 设计 |
| --- | --- |
| Sidebar | 固定左侧导航，按 Operations / Evidence / Growth / System 分组。 |
| Topbar | 当前模块、全局搜索、系统状态、设置入口。 |
| Main | 以表格、指标条、双栏详情、抽屉详情为主。 |
| Detail | 右侧 inspector 或独立详情页，展示对象全生命周期。 |
| Modal | 仅用于确认、短表单、批量动作预览。 |

## 11. 实施阶段

### Phase 0: 生产登录与启动硬化

目标：先保证后台可以安全地在公网生产环境对外服务，再扩展运营模块。

任务：

- 改造 `/api/auth/login`：zod 校验、统一错误、IP/username 双维度限流、失败登录记录。
- 增加 `admin_user_sessions`：登录创建 session，`/api/auth/me` 和 middleware 校验 session 有效，登出撤销 session。
- 增加强制改密：默认管理员首次登录必须改密，改密前只能访问 `/change-password` 和必要 auth API。
- 增加账号锁定与解锁：连续失败达到阈值锁定账号，Users & Access 提供解锁和踢下线。
- 增加 `admin_security_events` 与登录审计：记录新设备、多 IP、锁定、改密、登出、撤销会话。
- 强化 middleware：生产缺失强 `JWT_SECRET` 时拒绝启动或拒绝认证；生成 `x-request-id`；阻断常见扫描路径。
- 增加 Docker entrypoint：按 `ops:check-env -> db:init -> db:migrate -> admin:ensure -> supervisord` 顺序启动。
- 调整 supervisor 日志：容器部署输出到 stdout/stderr，web 与 worker 独立重启。
- 扩展 `/api/health`：展示数据库、worker、媒体、AI/PartnerBoost/Google Indexing 的 ok/degraded/unavailable。
- 明确公开路由白名单：SEO 页面、feed、sitemap、robots、open API 永远不要求后台登录。

验收：

- 未登录访问 `/admin` 跳转登录，未登录访问 `/api/admin/*` 返回 401。
- 生产环境默认 `JWT_SECRET` 或默认管理员密码会阻止服务启动。
- 连续失败登录会锁定账号，并在后台留下安全事件。
- 默认管理员首次登录必须改密，改密后才能进入后台。
- 登出、踢下线、账号禁用后旧 cookie 立即失效。
- Docker 容器冷启动后完成初始化并通过 `/api/health`。
- Worker 可独立重启，web 不因 worker 重启而不可用。
- 公开网站页面和 sitemap 在未登录状态下可访问、可抓取。

### Phase 1: 后台信息架构与总控升级

目标：让后台从“局部内容工具”变成“证据运营控制台”。

## 13. 生产上线验收清单

Bes3 后台管理系统要对公网生产服务，至少必须通过以下验收：

### 13.1 环境与启动

| 检查项 | 验收标准 |
| --- | --- |
| `npm run ops:check-env` | 生产环境无 error；默认 `JWT_SECRET`、短管理员密码、缺少 APP URL 会失败。 |
| `npm run db:init` | 能创建或补齐 schema、默认设置和管理员账号。 |
| `npm run db:migrate` | SQLite/PostgreSQL 均幂等；失败时启动中止。 |
| `npm run admin:ensure` | 可重建管理员账号，不在日志中长期暴露生产密码。 |
| Docker 启动 | 按预检、初始化、迁移、管理员、web、worker 顺序执行。 |
| `/api/health` | 返回 `status=ok`，包含 database、worker、media，外部依赖降级不误报主服务健康。 |

### 13.2 登录与权限

| 检查项 | 验收标准 |
| --- | --- |
| 登录成功 | 设置 `HttpOnly/Secure/SameSite=Lax` cookie，并写入 `admin_user_sessions`。 |
| 登录失败 | 返回统一错误，不泄露账号是否存在；写入 `admin_login_attempts`。 |
| 限流 | 同 IP 和同 username 触发阈值后被限制，多实例生产使用共享存储。 |
| 账号锁定 | 连续失败达到阈值后 `is_active=false` 或 lock 字段生效，Admin 可解锁。 |
| 强制改密 | bootstrap 管理员首次登录必须改密，改密前不能访问其他后台页。 |
| 会话撤销 | logout、禁用账号、踢下线、改密后旧会话不可继续访问 admin API。 |
| RBAC | Viewer 不能写，Evidence Ops 不能改密钥，Content/SEO Editor 不能改评分公式。 |

### 13.3 公开服务与运维

| 检查项 | 验收标准 |
| --- | --- |
| 公开页面 | `/products`、`/reviews`、`/guides`、`/deals`、`/sitemap.xml`、`/robots.txt` 未登录可访问。 |
| 后台保护 | `/admin/*` 未登录跳转 `/login`，`/api/admin/*` 未登录返回 401。 |
| Worker | Pipeline worker 可独立重启，后台能看到 worker 配置、心跳、失败任务。 |
| 持久化 | 数据库、媒体、备份目录挂载明确；生产推荐 PostgreSQL + S3/R2。 |
| 日志 | web、worker、auth、security event 均带 request id，可接入集中日志。 |
| 备份恢复 | `ops:backup-runtime` 和 `ops:restore-runtime` 有演练记录。 |

## 14. 成功标准

后台完成后，Bes3 应满足：

- 运营人员能判断每个商品是否可推荐。
- 运营人员能判断每条证据是否可信。
- 运营人员能解释每个共识分和买入窗口。
- 运营人员能处理断链、缺货、价格过期、错配、SEO 缺陷。
- 运营人员能从搜索意图推进到标签、证据回扫、pSEO 页面。
- 所有 public-facing 变化都有后台操作记录。
- 常规工作不依赖手工 SQL 和命令行脚本。

最终形态：Bes3 后台是一个面向 Evidence Graph 的运营系统，借鉴 autobb 的队列、监控、设置、风险和可操作列表架构，但服务于 Bes3 的硬核商品证据与程序化 SEO 业务闭环。
