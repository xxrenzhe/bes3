# Bes3 生产告警与上线 SOP

本文档补齐 Bes3 正式对外服务所需的最后一层运维闭环，基于当前约束：

- 对外正式域名：`https://bes3.com`
- HTTPS：由 Cloudflare 终止
- 部署平台：ClawCloud，手动从 GHCR 拉取镜像
- 数据库：第三方云 PostgreSQL
- 媒体与对象存储：第三方云对象存储
- 单容器单入口：容器内用 `supervisord` 管理 Web + Worker，对外只暴露 `80`

## 1. 发布前硬性门槛

发布前必须满足以下条件，否则不允许上线：

1. `npm run ops:preflight-release .env.production`
2. `JWT_SECRET` 已替换，且长度至少 32
3. `DEFAULT_ADMIN_PASSWORD` 已替换，且长度至少 16
4. `NEXT_PUBLIC_APP_URL=https://bes3.com`
5. `BES3_ALLOWED_HOSTS` 包含 `bes3.com`、`www.bes3.com`
6. `DATABASE_URL` 指向生产 PostgreSQL
7. `MEDIA_DRIVER=s3` 且对象存储参数完整
8. `BROWSER_PROXY_URLS_JSON` 为有效 JSON 数组；若启用强制代理抓取，代理池不能为空
9. `GOOGLE_INDEXING_ENABLED=true` 时，`GOOGLE_SERVICE_ACCOUNT_JSON` 完整有效
10. 至少完成一次真实商品全流程验证：联盟同步 -> 深抓取 -> 视频匹配 -> 证据提取 -> 页面发布

## 2. 一键预检命令

仓库提供统一预检命令：

```bash
npm run ops:preflight-release .env.production
```

它会串行执行：

1. `ops:check-env`
2. `ops:check-dependency-audit`
3. `ops:check-planv2-security`
4. `db:check-drift`
5. `lint`
6. `type-check`
7. `build`

如果要在运行中实例上追加健康检查：

```bash
BES3_PREFLIGHT_RUN_HEALTH=true \
BES3_PREFLIGHT_HEALTH_URL=http://127.0.0.1/api/internal/health \
BES3_PREFLIGHT_HEALTH_TOKEN="$JWT_SECRET" \
npm run ops:preflight-release .env.production
```

## 3. 标准上线流程

### 3.1 镜像准备

1. 确认 `main` 分支最新提交已经触发 GHCR 构建
2. 确认目标镜像存在：
   - `ghcr.io/xxrenzhe/bes3:prod-latest`
   - 或指定 `ghcr.io/xxrenzhe/bes3:prod-{commitid}`

### 3.2 服务器发布

在 ClawCloud 宿主机对应目录执行：

```bash
GHCR_USERNAME=<github-username> \
GHCR_TOKEN=<ghcr-token> \
BES3_APP_DIR=/srv/bes3 \
BES3_IMAGE=ghcr.io/xxrenzhe/bes3:prod-latest \
./scripts/deploy-ghcr.sh
```

该脚本会执行：

1. 镜像拉取
2. `check-runtime-env.js`
3. 数据库迁移
4. `docker compose up -d --no-build`
5. `/api/health` 健康探测

## 4. Cloudflare 与外部访问面

### 4.1 Cloudflare 侧要求

1. `bes3.com` 与 `www.bes3.com` 均指向 ClawCloud
2. 强制 HTTPS
3. 开启基础 WAF / Bot 防护
4. 对 `/api/internal/*` 不做公开暴露，必须仅通过内部 token 调用
5. 若使用 Cloudflare Cache Rule，不要覆盖应用返回的 HTML 缓存策略

### 4.2 对外路径要求

公开可访问：

- `/`
- `/categories/*`
- `/products/*`
- `/reviews/*`
- `/compare/*`
- `/deals/*`
- `/api/open/*`
- `/api/health`
- `/robots.txt`
- `/sitemap.xml`
- `/.well-known/security.txt`

不得公开入口：

- `/admin/*`
- `/api/admin/*`
- `/api/internal/*`
- `/api/auth/*`

## 5. 生产告警矩阵

### P0 / 立即处理

1. `/api/health` 返回非 `200`
2. `/api/internal/health` 返回 `503`
3. 数据库不可连接
4. Web 进程无法启动
5. 登录失效面积异常扩大

### P1 / 15 分钟内处理

1. Worker heartbeat 超过 2 分钟未刷新
2. `migration_history` 与迁移目录出现 drift
3. `admin_risk_alerts` 出现新的 `high` / `critical` 风险
4. `ops:check-dependency-audit` 出现非 allowlist 新告警
5. 联盟同步、深抓取、SEO 推送连续失败

### P2 / 当天处理

1. Google Indexing 长时间未成功
2. Link inspector 持续新增断链
3. 价格快照刷新延迟
4. Price alert notification 持续堆积

## 6. 每日运维检查

每天至少一次：

1. 检查 `/api/internal/health`
2. 检查后台 `Risk Center`
3. 检查后台 `Pipeline Runs`
4. 检查后台 `SEO Ops`
5. 检查后台 `Price & Value`

重点字段：

- `database.connected=true`
- `worker.enabled=true`
- `worker.heartbeatFresh=true`
- `migrations.pending=[]`
- `admin_risk_alerts` 无新增高优先级项

## 7. 备份与恢复责任边界

### 主备份源

1. PostgreSQL：由云数据库备份负责
2. 对象存储：由云对象存储版本控制 / 生命周期 / 跨区备份负责

### 仓库内补充备份

仓库脚本仍保留运行态兜底：

```bash
npm run ops:backup-runtime
BES3_RESTORE_CONFIRM=restore npm run ops:restore-runtime -- ./backups/<archive>.tar.gz
```

注意：

- 这不是云数据库和云对象存储的主备份替代
- 它主要用于本地卷、过渡期文件和演练恢复

## 8. 事故处理 SOP

### 8.1 Web 健康失败

1. `docker logs` 查看 `bes3-web`
2. 调用 `/api/internal/health`
3. 检查 `.env.production`
4. 检查 PostgreSQL 连通性
5. 如当前镜像有问题，回滚到上一个 `prod-{commitid}` 标签

### 8.2 Worker 卡死或队列堆积

1. 查看 `bes3-worker` 日志
2. 查看 `Pipeline Runs`
3. 确认代理池、AI key、联盟 token 是否过期
4. 必要时重启容器，让 worker 重新接管队列

### 8.3 发布后 SEO / 索引异常

1. 检查 `SEO Ops`
2. 确认 `GOOGLE_INDEXING_ENABLED`
3. 确认 `GOOGLE_SERVICE_ACCOUNT_JSON`
4. 手动执行：

```bash
npm run seo:index
```

## 9. 当前已接受的残余风险

当前唯一已登记残余风险：

- `npm audit` 中 `Next.js` 自带 `node_modules/next/node_modules/postcss@8.4.31`
- 状态：`moderate`
- 治理方式：`npm run ops:check-dependency-audit`
- 风险中心标识：`dependency_audit / next-bundled-postcss`

这条风险已纳入后台风险中心监控，不阻塞当前对外上线，但必须持续跟踪上游 Next.js 更新。
