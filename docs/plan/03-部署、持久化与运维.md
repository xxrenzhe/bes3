# 部署、持久化与运维

## 1. 本地媒体文件如何同步到生产

用户之前提出的关键问题是：

“媒体文件落本地持久化卷 `storage/media`，不写入仓库，那么同步到生产环境下呢？”

答案分两种部署模式：

### 1.1 单机模式

本地媒体不通过 Git 同步，而是通过服务器卷持久化。

实施方式：

1. 服务器创建目录：
   - `./data`
   - `./storage/media`
2. `docker-compose.yml` 把这两个目录挂到容器内：
   - `./data:/app/data`
   - `./storage/media:/app/storage/media`
3. 每次重建容器时，卷内文件继续保留，不会丢失。

迁移到新服务器时：

1. 同步代码仓库。
2. 同步 `.env.production`。
3. 用 `rsync` 或备份恢复工具同步 `data/` 和 `storage/media/`。

示例：

```bash
rsync -avz ./data/ root@new-server:/srv/bes3/data/
rsync -avz ./storage/media/ root@new-server:/srv/bes3/storage/media/
```

### 1.2 多实例模式

如果生产有多台应用实例，就不要继续用本地媒体卷，而要切到对象存储。

推荐：

- `MEDIA_DRIVER=s3`
- `MEDIA_PUBLIC_BASE_URL=https://cdn.example.com/bes3-media`

这样所有实例都读写同一个 Bucket，不需要再做文件同步。

## 2. Docker 部署

### 2.1 目录准备

建议生产目录结构：

```text
/srv/bes3
  ├── .env.production
  ├── docker-compose.yml
  ├── data/
  └── storage/
      └── media/
```

### 2.2 首次上线

```bash
mkdir -p /srv/bes3/data
mkdir -p /srv/bes3/storage/media
cd /srv/bes3
cp .env.production.example .env.production
docker compose build
docker compose run --rm bes3 node /app/scripts/check-runtime-env.js
docker compose up -d
```

如果服务器直接从 GHCR 手动发布：

```bash
GHCR_USERNAME=<github-username> \
GHCR_TOKEN=<ghcr-token> \
BES3_APP_DIR=/srv/bes3 \
BES3_IMAGE=ghcr.io/xxrenzhe/bes3:prod-latest \
./scripts/deploy-ghcr.sh
```

### 2.3 应用检查

上线后检查：

1. 访问 `/api/health`
2. 访问 `/login`
3. 用默认管理员登录后台
4. 进入 `Products` 页面，执行一次推广链接导入和完整流水线

## 3. SQLite 与 PostgreSQL 选型

### 3.1 SQLite

适合：

- 单机
- 单管理员
- 数据量中等
- 优先快速上线

注意：

- 只能单节点稳定使用
- 必须保证 `data/` 是持久化卷

### 3.2 PostgreSQL

适合：

- 多实例
- 更高并发
- 需要和对象存储配合扩展

切换方式：

1. 配置 `DATABASE_URL`
2. 保持其他代码不变
3. 重启应用即可

## 4. 发布流程建议

### 4.1 代码发布

1. 拉取最新代码
2. 更新 `.env.production`
3. `docker compose build`
4. `docker compose run --rm bes3 node /app/scripts/check-runtime-env.js`
5. `docker compose up -d`
6. 验证 `/api/health`

### 4.2 数据备份

单机 SQLite：

- 备份 `data/bes3.db`
- 备份 `storage/media/`

仓库已提供脚本：

```bash
npm run ops:backup-runtime
```

默认会把 `data/` 和 `storage/media/` 打包到 `backups/`。

恢复示例：

```bash
BES3_RESTORE_CONFIRM=restore npm run ops:restore-runtime -- ./backups/bes3-runtime-20260403-120000.tar.gz
```

PostgreSQL + S3：

- 备份 PostgreSQL
- 启用 Bucket 生命周期或对象存储版本控制

## 5. 运营工作流

后台的顺畅工作流建议按下面执行：

1. 先在 `Settings` 中配置 AI、联盟 Token、媒体驱动。
2. 到 `Products` 页面执行联盟同步，拉回商品池。
3. 对目标商品点击运行流水线，完成：
   - 抓链接
   - 抓商品
   - 抓媒体
   - 建库
   - 挖关键词
   - 生成评测
   - 生成对比
   - 写 SEO
4. 到 `Articles` 页面检查生成结果。
5. 到公开站点验证 `/reviews/...` 与 `/compare/...` 页面。

## 6. 上线前检查清单

1. `npm run type-check`
2. `npm run build`
3. `DEFAULT_ADMIN_PASSWORD` 已替换
4. `JWT_SECRET` 已替换
5. `NEXT_PUBLIC_APP_URL` 已替换成正式域名
6. 联盟 Token 已注入
7. AI Key 已注入

9. 已至少完成一次真实商品全流程导入验证
