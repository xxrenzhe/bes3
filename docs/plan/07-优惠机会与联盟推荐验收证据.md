# Bes3 优惠机会与联盟推荐验收证据

## 1. 验收结论

截至本次验收，`docs/plan/06-优惠机会与联盟推荐重构方案.md` 的核心要求已经在公开产品层落地。

当前判断：

- 已完成正式定位迁移：公开语义以“offers / promotions / best time to buy”为主，不再把“coupon”作为正式产品概念。
- 已完成主入口迁移：`/offers`、`/offers/[category]`、`/biggest-discounts` 已上线，`/deals` 仅保留兼容跳转。
- 已完成联盟准入收口：公开商品查询统一经过联盟可执行过滤。

## 5. 关键实现说明

### 5.1 公开 coupon 语义处理

当前代码保留了内部 ingestion 字段：

- `coupon_text`
- `coupon_type`

但这些字段已被限制在内部数据层和抓取 / pipeline 侧。

公开层处理方式：

- API 输出使用 `promotionSummary`，不直接暴露 `couponText`
- UI 展示使用 promotion 语义，而不是“领券 / 券后价”
- 如果原始文案带有 coupon / promo / voucher 词汇，会先清洗成中性促销描述

### 5.2 3 选 1 的最终收口点

最终推荐收口不是只发生在 offers 页面，而是分布在以下层：

- `src/lib/offers.ts`
- `src/lib/product-finalists.ts`
- `src/lib/commerce-intent.ts`
- `src/app/api/open/commerce/compare/route.ts`

其中 brand hub 曾是最后一个真实漏洞点，原因是它此前会把同品牌跨品类商品直接送入 finalists；现已修复。

## 6. 持续观察项

以下事项不构成当前验收失败，但需要持续观察：

- 参考价质量仍依赖联盟 payload 与落地页抓取质量，数据新鲜度不足时会被正确降级，但不会自动补齐缺失参考价。
- 内部数据库与 pipeline 仍保留 `coupon_text` / `coupon_type` 字段，这是为了兼容上游数据，不代表公开定位回退。
- `bd` 命令会持续提示 `.beads` 目录权限为 `0755`，建议后续按提示改成 `0700`，但这不影响本轮产品验收结论。
