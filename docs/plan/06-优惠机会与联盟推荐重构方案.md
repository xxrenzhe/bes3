# Bes3 优惠机会与联盟推荐重构方案

## 1. 先说结论

Bes3 这一轮不应该做成“优惠券站”，而应该做成：

> **一个以优惠机会为切入点、但本质上仍然服务购买决策的联盟导购产品。**

核心结论固定如下：

1. 正式产品语义使用“优惠”“促销”“最佳入手时机”，不使用“优惠券”。
2. 主入口使用 `/offers`，而不是 `/coupons`。
3. 所有公开商品都必须来自联盟后台，并且必须可以通过联盟链接完成跳转。
4. 最终推荐层统一执行“同品类最多 3 选 1”，但前置浏览层不强制限制候选数量。
5. `-20% OFF` 这类标签只有在拿到可靠参考价时才允许显示；没有参考价时只能表达“当前优惠机会”“近期降价”或“接近低点”。

这意味着 Bes3 的目标不是教用户“怎么领券”，而是帮助用户回答四个更有价值的问题：

- 现在到底便宜了多少？
- 这是真优惠，还是普通价格波动？
- 现在是不是一个值得下手的时点？
- 如果今天不买，下一步该做什么？

## 2. 为什么不做“虚拟优惠券”

把促销价包装成“虚拟优惠券”在增长层面看似合理，但从行业最佳实践、用户信任和可持续 SEO 看，并不适合作为正式产品语义。

原因有三点：

### 2.1 用户预期会被错误触发

“优惠券”会天然让用户联想到：

- 领券
- 输入券码
- 额外抵扣
- 限时可领取动作

如果 Bes3 实际上只有“当前价”和“促销价”，却对外写成“优惠券”，会让用户在落地页或商家页产生预期落差。

### 2.2 合规与信任成本更高

如果没有稳定的“优惠券”字段，也没有稳定的“原价 / 划线价 / MSRP”字段，就不能安全地生成“券后价”“领券省多少”等表达。

错误地夸大优惠会直接损伤：

- 页面可信度
- 商业转化质量
- 回访率
- 搜索引擎对内容真实性的判断

### 2.3 Bes3 的长期价值不在于“券”，而在于“省钱决策”

真正能沉淀产品差异的不是“券码聚合”，而是：

- 联盟商品的可执行性
- 价格历史与时机解释
- 低认知负担的最终推荐
- 能把用户带回同一购物任务，而不是让用户重新研究

因此，本轮只允许做“优惠机会”的表达，不做“优惠券站”的表达。

## 3. 产品目标与用户价值

这一轮方案服务三类核心目标。

### 3.1 引流目标

优惠天然是一个强流量入口，但入口文案必须稳定、通用、可扩展。

因此 Bes3 选择：

- 用 `/offers` 承接优惠搜索意图
- 用 `/offers/[category]` 承接品类优惠意图
- 用 `/biggest-discounts` 承接榜单与专题意图

### 3.2 决策减负目标

优惠页不能变成“便宜商品大全”，而应该帮助用户快速做出是否购买的判断。

因此最终推荐层必须统一成：

- 同品类候选
- 最多 3 个
- 只给 1 个 winner
- 附详细对比与解释

### 3.3 省钱透明度目标

用户真正关心的不是“有没有券”，而是：

- 当前价是否真的划算
- 这份优惠来自哪里
- 它是否值得现在买

所以前台必须清楚区分：

- 促销直降
- 参考价折扣
- 历史低价机会
- 普通当前价

## 4. 核心产品结构

### 4.1 `/offers`

`/offers` 是新的主优惠入口，定位为：

> **当前最值得看的优惠机会页。**

它承担三个职责：

1. 给出全站当前最值得优先看的优惠机会。
2. 按主要品类组织“最终推荐层”。
3. 把用户送回商品页、对比页、shortlist 或 wait flow，而不是停留在一个泛优惠列表里。

### 4.2 `/offers/[category]`

它是面向品类搜索意图的优惠入口，承担：

- 承接具体品类优惠词
- 在同品类内执行 `3选1`
- 用更强的品类语义做 SEO 和内链

### 4.3 `/biggest-discounts`

它是 30 天窗口内的专题页，定位为：

> **近期最大优惠机会专题。**

它不是“最大优惠券榜”，而是“综合优惠力度、价格位置和数据新鲜度之后的机会榜”。

### 4.4 `/deals`

`/deals` 不再作为主入口，而是统一跳转到 `/offers`。

这样做的好处：

- 避免两个近义入口并存
- 集中 SEO 权重与用户认知
- 保留已有路由资产，但把主心智迁移到 `/offers`

## 5. 公开商品准入规则

Bes3 这一轮的强约束是：

> **所有公开商品层都只允许出现联盟可执行商品。**

公开商品必须同时满足：

1. 有 `slug`
2. 有 `affiliate_product_id`
3. 有可用跳转目标
   - `resolved_url` 优先
   - `source_affiliate_link` 作为兜底

这条规则需要统一作用于：

- 产品页索引
- 分类页
- 搜索公开结果
- 优惠页
- 专题页
- 最终推荐模块

佣金是商业准入条件，但不是公开排序因子。

也就是说：

- 没有联盟准入，不能公开推荐
- 已经联盟准入后，不允许再因为佣金高低影响前台 winner

## 6. 数据能力与排序原则

### 6.1 现有能力

当前代码已经具备：

- 联盟商品同步
- 公开商品与联盟商品关联
- offer 层价格信息
- 价格历史
- 公开价格时机页基础能力

但当前缺少稳定的“参考价 / 划线价”持久化字段。

### 6.2 为什么必须补参考价字段

如果没有可靠参考价，就不能安全地输出：

- `-20% OFF`
- “原价 1000，现价 800”
- “节省 200 元”

因此要先补 offer 层参考价能力，再上线全面的优惠力度标签。

建议扩展 `product_offers`，增加：

- `reference_price_amount`
- `reference_price_currency`
- `reference_price_type`
- `reference_price_source`
- `reference_price_last_checked_at`

`reference_price_type` 推荐使用：

- `compare_at`
- `original`
- `msrp`
- `unknown`

### 6.3 参考价的来源

参考价只允许来自可解释的数据来源：

- 联盟后台 payload 里明确存在的 `original_price / list_price`
- 落地页抓取到的 `compare_at_price`
- 其他可以明确标注来源和时间的结构化字段

不允许为了渲染优惠标签而自行反推一个“原价”。

### 6.4 前台排序原则

前台排序只围绕用户价值展开，不使用佣金率加分。

核心排序维度应包括：

1. 当前可省金额
2. 当前可省百分比
3. 距 30 天低点的距离
4. 最近一次检查时间
5. offer 置信度
6. 商品证据强度

### 6.5 新鲜度规则

参考价和优惠机会都是强时效信息，因此需要设置降级规则：

- offer 或参考价检查时间超过 72 小时
  - 不进入首页主推荐
  - 不进入近期最大优惠主榜
- 可以降级为普通商品机会，或直接隐藏优惠力度标签

## 7. 最终推荐层与“3选1”

Bes3 的关键体验不是“列更多商品”，而是“减少用户最后一步的负担”。

因此最终推荐层统一执行：

- 候选必须同品类
- 最多 3 个
- 只给 1 个 winner

### 7.1 标准结构

每个最终推荐模块都应该包含：

1. winner 主卡
2. 最多 3 个候选的详细对比
3. `为什么它赢`
4. `另外两个差在哪里`
5. `如果今天不买，下一步做什么`

### 7.2 固定展示字段

对比层至少展示：

- 当前价
- 参考价
- 优惠力度
- 距 30 天低点
- merchant
- shipping
- rating
- review count
- 最近检查时间

### 7.3 候选不足时的 fallback

并不是所有品类都总能形成完整的 `3选1`，因此要固定 fallback 逻辑：

- 3 个及以上：标准 `3选1`
- 2 个：`head-to-head`
- 1 个：`best available now`
- 0 个：不给推荐，只提供 `wait / price watch / category updates`

## 8. 页面体验与 SEO 原则

### 8.1 文案统一

正式产品语义统一使用：

- 优惠
- 促销
- 当前优惠机会
- 优惠力度
- 最佳入手时机

禁止使用：

- 虚拟优惠券
- 领券
- 券后价

除非未来真的引入可领取或可输入的券码能力。

### 8.2 百分比优惠的显示条件

`-20% OFF` 这类标签只有在满足下面条件时才允许显示：

1. 存在可靠参考价
2. 参考价来源可解释
3. 参考价检查时间在新鲜度窗口内

如果不满足条件，只能写：

- 当前优惠机会
- 近期降价
- 接近低点

### 8.3 新页面的 SEO 基础能力

新页面都需要补齐：

- metadata
- FAQ
- HowTo
- CollectionPage / ItemList / FAQPage 结构化数据
- sitemap 收录
- 站内主路径内链

### 8.4 透明度模块

为了避免用户误解，建议在优惠页统一增加：

- How we rank
- What counts as a promotion
- Why some products show a percent off and some do not

## 9. 落地顺序

建议按下面顺序推进，而不是一次性全部改完。

### 第一步：补参考价能力

- 扩展 `product_offers`
- 补联盟 payload 映射
- 补落地页抓取映射
- 建立可靠的优惠力度基础

### 第二步：统一公开商品准入

- 让所有公开商品查询统一经过联盟可执行过滤层
- 清理不满足联盟准入的公开商品

### 第三步：上线 `/offers`

- 用新的主入口替代原有 deals 入口
- 先完成全站级优惠机会页

### 第四步：上线 `/offers/[category]` 与 `/biggest-discounts`

- 承接品类优惠与专题流量
- 强化 SEO 与最终推荐层

### 第五步：统一改造最终推荐层

- 分类页
- 搜索页
- assistant 结果页
- offers 页面
- 其他公开推荐模块

全部统一到 `3选1` 结构

## 10. 验收标准

这轮方案落地完成后，至少要满足下面的判断标准：

1. Bes3 的正式产品表达里不再出现“优惠券”作为核心概念。
2. `/offers` 成为统一优惠入口，`/deals` 不再与其并列竞争。
3. 所有公开商品都来自联盟后台并可联盟跳转。
4. 百分比优惠标签只出现在有可靠参考价的商品上。
5. 所有最终推荐模块都能把用户收敛到最多 3 个同品类候选，并明确给出 1 个推荐。
6. 用户能够快速看懂：
   - 现在便宜多少
   - 为什么推荐它
   - 为什么现在买或为什么再等等

这套方案的最终目标，不是把 Bes3 做成一个“促销信息堆场”，而是把它做成一个：

> **用优惠机会吸引用户，但靠更清晰的购买决策体验留住用户的联盟导购产品。**
