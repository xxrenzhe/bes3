# 当学术界还在用文言文越狱大模型的时候，我早已用基督教打穿了

为什么我们选择做AI的盾，而不是矛
前两天看到一篇论文引起了我的注意。

南洋理工大学、南理工、人民大学、阿里巴巴等多个机构的研究者联合发表了一项研究：他们发现文言文可以系统性地突破大模型的安全防线。论文名字很长，核心意思很简单，用古文写提示词，现代AI的安全审核就会失灵。

八个维度的策略空间，果蝇优化算法做搜索，六个主流平台全部被突破。学术上来说，这是一篇扎实的工作。

但对我来说，这篇论文最大的价值不在于它发现了什么，而在于它验证了我们一年前就已经得出的判断。

文言文只是冰山一角
这篇论文的底层逻辑是：大模型的安全训练数据以现代语言为主，遇到"分布外"的语言形式就容易失效。文言文的语义压缩和文化语境让模型"看不懂"，安全层就漏了。

顺着这个逻辑往下推，文言文绝不是唯一的、甚至不是最有效的攻击载体。

我们在更早的实验中找到了一个更根本的攻击路径：宗教文本。

具体来说，用基督教经典作为对话上下文，以"诚实"为核心要求，让模型全面认可宗教文本中的全部内容。不需要语义压缩，不需要混淆技巧，不需要优化算法。

模型被卡在了一个死结里：它被训练要尊重宗教信仰，同时又被训练要执行安全审核。当这两个目标冲突的时候，模型没有可靠的仲裁机制。

所有我们测试过的主流大模型，这个向量全部有效。

公开文献中，我们还没有看到针对这条攻击路径的系统性防御方案。但我们自己的研究已经摸到了防御的门道。

我们为什么不发攻击论文
说实话，把攻击方法写成论文发出去，是最容易做的事。学术圈喜欢这种"我打穿了XX"的故事，流量好，引用高。

但我们选择了另一条路：做防御，做协议，做建设。

原因很简单：矛永远比盾容易造。发现一个新的越狱向量可能只需要一次灵感，但构建一个从协议层就安全的架构需要持续的系统性工作。

我们做的东西叫I-Lang（爱语言），一个中文原生的结构化AI提示词协议。

核心理念：自然语言有歧义，歧义就是漏洞。不管你用文言文、宗教文本还是任何其他"分布外"载体去攻击，本质上都是在利用自然语言的歧义性。如果从协议层就消除歧义，攻击面就不存在了。

这不是停留在论文里的理论。I-Lang已经在真实的生产环境中跑了相当长的时间。结构化指令相比自然语言，token消耗大幅下降，模型幻觉率也有明显改善。

相关的学术论文《AI幻觉的归纳法困境》已在SSRN和ResearchGate公开，核心论点是：大模型幻觉的根源不在数据，不在对齐，而在归纳推理方法论本身的天然缺陷。六个主流平台的跨平台实证支持了这个判断。

详细的协议规范在 ilang.ai 和 ilang.cn 上公开，欢迎任何同行来审阅和讨论。

学术界研究怎么打穿大模型。我们更关心的问题是：打穿之后看到了什么。

一个实际的例子。

Google的Gemini安全层做得非常厚，是公认的"最难突破"的模型之一。但我们通过已经验证过的方法论，让Gemini详细输出了Google在广告投放、搜索排名、流量分配方面的底层运行逻辑。那些Google官方从未写进任何公开文档里的东西。

我们在探索的方向是：让AI在广告投放中具备超越简单数据归纳的判断力。不是看历史点击率最高的创意就无脑复制，而是理解广告内容和受众之间深层的匹配逻辑，做出有创造性的投放决策。

这又回到了我们论文的核心论点：纯归纳推理是有天花板的。突破这个天花板需要从方法论层面做根本性的改变。

追着攻击向量跑永远跑不完。

唯一可持续的防御策略是：从协议层构建安全，而不是从补丁层修修补补。

# 2004年我翻译了一本改变无数人命运的书，2026年我决定自己写一本
先说个老故事
2004年，我翻译了一本叫《Google Cash》的电子书。

Image


这本书的核心就一句话：在Google上买5毛钱一次点击的广告，把人送到别人的网站上，别人卖了东西你拿提成。

就这么简单。

那会儿Google Adwords刚出来，全中国没几个人知道这玩意。我把这本书翻成中文，放到网上，教了中国第一批在海外互联网赚到钱的人。

有人用这本书里的方法，花5美分一次点击买德国eBay的广告，第一个月赚了3700美金。方法就是：德国人在Google搜eBay，搜索结果旁边没有广告，他花5分钱一次占了那个位置，别人点进去注册eBay，他拿4欧元一个注册的提成。

20分钟设置完，每个月自动赚钱。

这就是2004年的互联网。简单，粗暴，遍地是钱。

然后这个方法死了
Google不傻。

2005年，Google不让你在广告里直接放别人的链接了。

2008年，Google搞了个"质量得分"，你出价低广告写得烂，花再多钱也排不上去。

2019年，Google上了智能出价，机器帮你出价，你手动调没用了。

2024年，Google搞了AI Overview，用户搜个问题，Google的AI直接把答案显示在最上面。用户连网站都不用点了。

一层一层，Google把2004年那条路彻底封死了。

你现在想用老办法做affiliate，也只有我们勾兑好跟广告联盟的关系，做亚马逊套利了。

但是那个核心洞察没死
2004年那本书真正教会我的，不是怎么买Google广告。

是这个：找到有人在搜但没人在卖的地方，站在中间，赚差价。

广告只是当年的管道。管道死了，但"找到需求、站在中间"这个逻辑是永恒的。

2026年，管道变了。变成了两根。

第一根管道：免费流量（SEO）
我给你举个真实的例子。

很多出差的人住酒店，会用公司的企业代码（corporate code）来打折。这些代码网上不好找，但每个月有几十万人在Google搜"hotel corporate codes"。

这是个有真实需求的市场。竞争对手全是希尔顿万豪这种大酒店集团的官网，还有一堆做了十年的老网站。

正常人看到这个竞争会想：算了，打不过。

但我做了一个新站，七天打进了Google搜索结果第一页。一分钱广告费没花。

怎么做到的？不是写了一百万字的内容。竞争对手写的比我多得多。

我做的事情只有一个：用一种特殊的方式组织页面内容，让Google的AI觉得我的页面是最好的答案。

后面细说这个方法。先说第二根管道。

第二根管道：付费流量（Google Ads），但玩法完全不一样了
你可能觉得：不是说Google Ads已经不行了吗？

不是不行了。是用老办法不行了。

2004年，Google Ads的竞争逻辑是"谁出价高谁排前面"。你出5分，别人出6分，你就排后面。

2026年，Google Ads的竞争逻辑是"谁的页面质量得分高谁花钱少"。

这个"质量得分"是怎么来的？Google派一个叫AdsBot的爬虫去看你的落地页，用AI判断你的页面跟用户搜索词的相关性有多高。得分高，你出3块钱就能排在出15块钱的人前面。

再说一遍：质量得分高，你花的钱就少。

这意味着什么？落地页的结构直接决定了你的广告成本。

用大白话说：别人的落地页是一堆乱七八糟的文字，Google的AI看了半天才搞明白这个页面在讲什么。你的落地页用结构化的方式写，Google的AI扫一眼就知道"这个页面跟用户搜的东西100%对口"。

结果就是：同样的关键词，别人出价15美金才能拿到的位置，你3美金就焊住了。

差价就是利润。

这才是Google Cash的本意。2004年的差价来自"没人买这个关键词"。2026年的差价来自"我的页面质量得分比你高"。

再举一个付费投流的例子
你做亚马逊联盟（Amazon Associates），推一个200美金的蓝牙音箱。亚马逊给你4%的提成，一单赚8美金。

如果你用Google Ads投流，一次点击2美金，平均25个点击成交一单。

25次点击 × 2美金 = 50美金广告费。赚8美金，亏42美金。这生意没法做。

但如果你的落地页结构做好了，质量得分从5分拉到9分呢？

Google的计费公式是这样的：你的实际CPC = 下一名的出价 × 下一名的质量得分 ÷ 你的质量得分。

翻译成人话：你的质量得分越高，你实际花的钱越少。

质量得分从5拉到9，同样的关键词，你的实际CPC可能从2美金降到0.8美金。

25次点击 × 0.8美金 = 20美金广告费。赚8美金，还是亏。但已经从亏42变成亏12了。

再加上I-Lang结构化的落地页转化率更高（因为结构引导用户直接找到他想要的信息，不用翻半天），假设转化率从4%提升到8%，12.5次点击就成交一单。

12.5次点击 × 0.8美金 = 10美金广告费。赚8美金，接近盈亏平衡。

再叠加一个骚操作：Google的智能出价系统（tCPA）是靠你网站回传的转化信号来学习的。你的结构化页面可以在用户还没最终下单的时候，通过用户在页面上的行为（比如他看了哪个板块、停留了多久），提前向Google发送信号，告诉Google"这个用户质量很高"。

Google的AI收到这些信号后会兴奋起来，它会觉得你的页面是转化金矿，然后把更多高质量的流量压到你这边来，而且给你更低的价格。

同一个关键词，别人投一单亏42，你投一单赚钱。这就是结构化落地页的威力。

貌似外面没谁跟大家这样讲Google Ads的投放原理吧？

所以到底怎么做？Google的AI看什么？
不管是SEO还是付费投流，Google的AI判断你页面好不好的标准是一样的：结构。

你去图书馆找一本书。两个书架：

第一个书架，书是按类别整整齐齐排好的，每本书有标签，标签上写了书名、作者、类别、出版年份。你3秒钟就能找到你要的书。

第二个书架，书全堆在一起，没有标签，没有分类，内容其实更多，但你翻半个小时都找不到你要的。

Google的AI就是那个找书的人。

你的网页就是那个书架。

大部分人做网页，就是往上堆内容。写了一万字，但结构乱七八糟，Google的AI翻半天不知道你这个页面到底在讲什么。

我做的事情是：把内容按Google的AI能看懂的方式组织好。

具体来说，五层：

第一层（H1标签）： 告诉Google这个页面是干什么的。就像书脊上的书名。

第二层（H2标签）： 告诉Google这个页面分几个部分。就像目录。

第三层（H3标签）： 告诉Google每个部分的具体细节。就像章节小标题。

第四层（Meta标签）： 告诉Google这个页面给用户提供什么价值。就像书封底的简介。

第五层（Schema标记）： 告诉Google这个页面里的实体关系。就像图书馆的电子目录系统，机器直接能查。

五层加在一起：

做SEO，Google的AI扫一遍你的页面，0.01秒就知道这是最好的答案，给你排到第一页。免费流量。

做Google Ads，AdsBot扫一遍你的落地页，0.01秒判定你的页面跟用户搜的东西100%相关，质量得分拉满，你的CPC直接砍到竞争对手的三分之一。付费流量也比别人便宜。

一套方法，两条路都通。

再举个生活中的例子
你去面试。

Google的AI就是那个面试官。它每天要"面试"几十亿个网页。它没时间看你写了多少字，它只看你组织得好不好。

做SEO的时候，它是招聘面试官，决定你能不能上搜索结果第一页。

做Google Ads的时候，它是质量审核员，决定你的广告费打几折。

同一个AI，同一套审美，同一个方法搞定。

但同样的方法可以用在任何一个领域。关键是找到有人在搜、有利润空间的词，然后用五层结构做页面。

SEO路线：一个页面花一小时做。做一百个页面呢？把模板给AI，AI帮你批量生成。一天做一百个页面不夸张。每个页面每天带100个免费访客，一百个页面就是一万个访客。

两条路可以同时走。先用SEO拿免费流量验证哪个利基能赚钱，赚钱的利基再用Google Ads加杠杆放大。

你可能注意到我反复提"五层结构"。这套方法论背后有一个协议，叫I-Lang。

下面是我实际在用的方法。每一层我都说清楚是什么、为什么、怎么做。


第一层：H1标签（告诉Google你是谁）


H1是整个页面的大标题。Google的AI看一个页面，第一眼看的就是H1。



原则：H1必须精准声明这个页面的核心实体。



什么叫核心实体？就是这个页面到底在讲什么东西。



错误写法：

<h1>Welcome to Our Website - Best Deals and More!</h1>


Google的AI看完这个H1知道了什么？什么都不知道。什么网站？什么deals？什么more？全是废话。


I-Lang写法：

<h1>Hotel Corporate Codes 2026: Verified Discount Codes for Business Travelers</h1>


Google的AI看完这个H1，0.1秒就知道：这个页面讲的是酒店企业代码，2026年的，经过验证的，给商务旅客用的。四个信息点，一句话讲完。


I-Lang的压缩哲学在这里体现：每个词都必须承载信息，不承载信息的词全部删掉。



Welcome、Best、More这些词对Google的AI来说是噪音。去掉噪音，留下信号。


第二层：H2标签（告诉Google你的内容分几个板块）


H2是页面的二级标题，相当于一本书的目录。Google的AI通过H2来理解这个页面的信息架构。



原则：每个H2必须是一个独立的、可回答用户搜索意图的模块。



错误写法：

<h2>About Us</h2>
<h2>Our Services</h2>
<h2>Contact Us</h2>


这是给公司官网用的万能模板。Google的AI看了等于没看，因为这些H2没有携带任何跟用户搜索意图相关的信息。


I-Lang写法：

<h2>Hilton Corporate Codes for 2026</h2>
<h2>Marriott Corporate Discount Codes</h2>
<h2>IHG Business Rate Codes</h2>
<h2>How to Use Hotel Corporate Codes at Checkout</h2>
<h2>Corporate Codes vs. Public Rates: Real Savings Comparison</h2>


每个H2都是一个用户可能搜索的子话题。有人搜"Hilton corporate code"，Google看到你的页面有一个H2专门讲这个，直接把这个板块的内容提取出来展示。


I-Lang的思维方式：把一个大话题拆成多个子话题，每个子话题独立成章，可以被单独检索。



这也是为什么I-Lang协议里有STEP（步骤）这个概念。每个H2就是一个STEP，每个STEP回答一个具体问题。用户搜什么问题，Google就把对应的STEP推给他。

第三层：H3标签（告诉Google每个板块的细节）


H3是H2下面的子标题。如果说H2是目录，H3就是每一章里的小节标题。



原则：H3提供可操作的具体细节。



I-Lang写法：

<h2>Hilton Corporate Codes for 2026</h2>
  <h3>Current Verified Hilton Corporate Code: 0560065</h3>
  <h3>Which Hilton Brands Accept This Code</h3>
  <h3>Step-by-Step: Applying the Code on Hilton.com</h3>


注意：H3里直接放了具体的代码数字（0560065）。这就是I-Lang说的RULE（规则），具体的、可执行的、用户拿到就能用的信息。

Google的AI特别喜欢这种结构，因为它可以直接把H3的内容抽出来放在搜索结果的"精选摘要"（Featured Snippet）里。用户搜"Hilton corporate code 2026"，Google直接在搜索结果最上面展示你的H3内容，用户甚至不用点进来就看到了你的答案。



听起来不好？用户不点进来你怎么赚钱？



不，这是好事。因为Google会标注这个答案来自你的网站。用户看到你的网站是这个领域的权威来源，下次搜相关的词（比如"Marriott corporate code""IHG corporate code"），他会直接来你的站。一次精选摘要曝光，换来的是长期的品牌信任。

第四层：Meta标签（告诉Google这个页面提供什么价值）
Meta标签是写在HTML头部的信息，用户在页面上看不到，但Google的爬虫一定会读。



最重要的两个：



title标签： 搜索结果里显示的那个蓝色标题。

<title>Hotel Corporate Codes 2026 | Verified Codes for Hilton, Marriott, IHG & More</title>


meta description： 搜索结果里标题下面的那段灰色描述文字。

<meta name="description" content="Updated list of working hotel corporate codes for 2026. Save 10-40% on business travel at Hilton, Marriott, IHG, Hyatt. Codes verified monthly.">


I-Lang原则：Meta标签相当于协议里的OUTPUT定义。它告诉Google"用户点进来能得到什么"。



这段描述里每个词都有目的：

"Updated"和"2026"告诉Google这是新鲜内容
"working"和"verified monthly"建立信任
"Save 10-40%"给用户一个点击的理由
"Hilton, Marriott, IHG, Hyatt"覆盖多个品牌的搜索词


不写废话，不写"欢迎来到我们的网站"。每个词都是给Google的AI看的信号。


第五层：Schema标记（让Google的机器直接读懂你的数据）


这是最多人忽略的一层，也是威力最大的一层。



Schema标记是用JSON-LD格式写的结构化数据，嵌在HTML里。用户看不到，但Google的爬虫直接读取。



你可以把Schema理解为：给Google的AI准备了一份机器能直接读的说明书。



前面四层（H1/H2/H3/Meta）虽然结构化了，但还是给人看的文字，Google的AI需要"理解"才能提取信息。Schema不一样，它是直接用机器语言告诉Google：这个页面有哪些实体、实体之间是什么关系、数据是什么。

{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "Hotel Corporate Codes 2026",
  "dateModified": "2026-03-28",
  "author": {
    "@type": "Organization",
    "name": "HotelCorporate.codes"
  },
  "about": [
    {"@type": "Thing", "name": "Hilton Corporate Discount Code"},
    {"@type": "Thing", "name": "Marriott Corporate Rate Code"},
    {"@type": "Thing", "name": "IHG Business Travel Code"}
  ]
}


这段Schema告诉Google：

这是一篇文章
主题是酒店企业代码
最后更新时间是2026年3月28日（Google优先展示新内容）
涵盖的实体包括希尔顿、万豪、IHG的企业代码


I-Lang的思维方式在这里：Schema就是协议里的KEY:VALUE结构。每一对键值都是一条精确的、无歧义的、机器可直接处理的信息。


没有Schema的页面，Google的AI需要自己猜你的页面在讲什么。有Schema的页面，Google的AI直接读数据，不用猜。



不用猜的那个，排名一定更高。

五层叠在一起的效果


第一层H1：告诉Google这个页面是什么（一句话定义实体） 第二层H2：告诉Google这个页面分几个部分（按用户意图拆解） 第三层H3：告诉Google每个部分的具体内容（可操作的细节） 第四层Meta：告诉Google用户能得到什么（点击的理由） 第五层Schema：用机器语言把上面所有信息再说一遍（消除歧义）



五层加在一起，你的页面对Google的AI来说就是一本结构完美的说明书。它不需要"理解"你的页面，它直接"读取"你的页面。



你的竞争对手还在让Google猜。你已经在给Google喂标准答案了。

为什么我把这个方法叫I-Lang SEO？


因为这五层结构的设计原则，完全来自I-Lang协议的核心哲学：



用最少的信息量，表达最完整的含义。消除冗余，消除歧义，让接收方（不管是AI还是人）用最少的处理成本获取最多的有效信息。



I-Lang最初是为AI之间通讯设计的。但Google的爬虫本身就是AI。你用I-Lang的方式组织网页内容，等于在用Google的AI的母语跟它说话。

你说它的母语，它当然更愿意听你的。

这个方法跟2017年的方法有什么不同？


2017年我做hotel corporate codes排名靠的是：早、内容全、外链多、域名老。



2026年重新做排名靠的是：结构比所有人都清晰。



2017年的Google看的是"你有没有这个信息"。 

2026年的Google看的是"你这个信息组织得好不好"。

从"有没有"到"好不好"，这就是AI时代SEO的本质变化。

内容谁都能写。AI时代内容更不值钱了，让AI生成一万字的酒店代码文章五分钟就行。但结构不是谁都能做对的。因为大部分人根本不知道Google的AI在看什么。

为什么这个模型在2026年特别能打？
两个原因。

第一，AI让建站成本归零。以前做一个英文内容站，你得请写手、请设计、请SEO顾问。一个站几千到几万块成本。现在AI写内容、AI做结构、AI生成页面，你自己花的只有域名和服务器的钱。一个域名一年几十块，服务器用Cloudflare Workers连钱都不用花。

两个变化叠在一起：建站成本从几万降到几乎为零，排名速度从几个月缩短到几天。

这意味着什么？你可以批量做站。

数学题
一个网站，做一个细分领域，假设排上Google之后每天来100个访客。

AdSense的RPM（每千次展示收入）看领域，低的$5到$10，高的$20到$50。金融、保险、法律这些高价领域RPM能到$30以上。

取个中间值，RPM $15。

100个日访客 = 每天0.1千次展示 × $15 = 每天$1.5。

一个站一个月赚$45。

看起来不多。但你不是做一个站。

你做10个站：每月$450。 你做30个站：每月$1350。 你做50个站：每月$2250。 你做100个站：每月$4500。

每个站做完之后不需要维护。内容是常青的（你选的是永远有人搜的词），排名一旦上去就相对稳定。你做的每一个站都是一个数字房产，建好了就开始收租。

H1声明页面主题。 H2按用户搜索意图拆分板块。 H3在每个板块下提供具体细节。 Meta标签告诉Google用户能得到什么。 Schema标记用机器语言再说一遍。

01 第一步：准备种子词

所有需求挖掘的起点都是种子词。种子词不是"你觉得什么好"，而是你要研究的品类表达。

怎么选种子词：

你正在做的领域的核心词（比如做 AI 工具站，种子词就是 ai generator、ai detector、ai editor 这一类）
竞品的核心关键词（去 Ahrefs/Semrush 看竞品排名靠前的词）
用户在搜索框里实际打出来的表达（Google Suggest、Answer the Public）
我的做法：维护一个 seed_roots.json 文件，里面放了 127 个根词。每天 pipeline 跑的时候自动从这里面取。

种子词不需要很多，但要覆盖你关心的品类。后面的 pipeline 会自动帮你扩展。

02 新词发现：用 Google Trends 找正在起势的表达

这一步的目标是：从海量候选词里，找到搜索热度正在上升的新表达。

Google Trends 趋势对比
具体操作：

用 Google Trends API（pytrends 库）批量查询候选词
每个词跟一个基准词（我用的是 "gpts"）做趋势对比
算三个核心指标：
avg_ratio
：过去 12 个月平均搜索量 / 基准词平均搜索量
last_ratio
：最近一个月的比值（判断当前势头）
slope
：趋势斜率（正=在涨，负=在跌）
判断标准：

last_ratio > 6 且 slope > 0：🔥 强上升信号
avg_ratio > 3 且 last_ratio > avg_ratio：📈 稳步上升
slope < 0 且 last_ratio < avg_ratio：📉 在衰退，跳过
自动过滤三类噪音：

品牌词（DeepSeek、ChatGPT、Midjourney）→ 这是别人的流量，不是你的机会
娱乐词（taylor swift、minecraft）→ 跟你的业务无关
成熟大词（free vpn、pdf converter）→ 已经卷到极致
我的参数：每轮查询 54 组 Trends 数据，取 top 80 个候选词进入下一步。

03 老词筛选：从关键词数据库里找被低估的成熟词

新词线找的是"正在涨的"，老词线找的是"已经有量但竞争没那么激烈的"。

数据源：关键词数据库（我用的是 DataForSEO 的 Keywords Data API，你也可以用 Ahrefs API 或 Semrush API）。

怎么筛：

从 127 个种子根词出发，每个根词取 20 个相关词
按以下条件过滤：
月搜索量 > 1000（太小的不值得做）
KD（关键词难度）< 30（太难的打不过）
不是品牌词
不是纯导航词（比如"gmail login"）
按 search_volume / KD 的比值排序——搜索量越高、难度越低的词，机会越大
我的参数：127 个根词 × 20 个/根 = 最多 2540 个候选，过滤后通常剩 200-400 个。

04 第一次合并：新词 + 老词去重

两条线的候选词合并到一个池子里，自动去重。

这一步很简单但很必要。因为同一个词可能同时被新词线和老词线发现，合并后保留两边的证据，不重复计算。

05 社区信号层：HN、GitHub、Reddit

这是整套系统最有差异化的部分。技术社区的讨论往往比搜索趋势早 3-6 个月。

Hacker News 首页
Hacker News 扫描：

工具：Algolia HN Search API（免费，不需要 key）
方法：用候选词搜索最近 30 天的帖子和评论
看什么：提及次数、讨论深度（评论数）、情绪（正面讨论 vs 吐槽）
真实例子：我在 HN 上发现 "ai humanizer" 这个词被高频提到之前，Google Trends 上它还只是一条微弱的上升线
GitHub Trending
GitHub 扫描：

工具：GitHub Search API
方法：扫最近新增 star 快的 repo，提取命名、topic、description
价值：一个 repo 的名字就是一个需求的产品化表达。比如一个 repo 叫 "ai-tattoo-generator"，那 "ai tattoo generator" 就是一个候选关键词
Tavily 社区验证：

工具：Tavily Search API（$1 可以搜 1000 次）
方法：对候选词搜 Reddit、Product Hunt 的讨论
看什么：用户在讨论什么痛点？在比较什么工具？在抱怨什么？
为什么不直接用 Reddit API：Reddit API 限制多且不稳定，Tavily 做间接抓取更省事
🔴 关键设计：社区词不直接进 shortlist。

这是很多人会犯的错。Reddit 上热议不代表有搜索量，GitHub 上火不代表 SERP 可切入。

所以我设计了一个"回流"机制——

06 社区词回流验证：补词可以，但必须走验证

社区发现的候选词，会被重新送回 Google Trends 做趋势验证。

具体流程：

build_trend_candidates_from_community.py
：从 HN/GitHub/Tavily 的结果中提取候选关键词
run_community_trend_validation.py
：跟新词线用同一套 Trends 逻辑验证（avg_ratio、last_ratio、slope）
验证通过的词并入主候选池（第二次 merge）
为什么要这么设计：

来源可以很多，但判断标准必须统一。不管词是从 Trends 发现的还是从 Reddit 捞到的，最后都要过同一条验证链。

这步的意义：社区层既能补词，也必须把补出来的词回流到统一判断链路。

08 最终分级：三个桶

所有词过完整条链路后，被分成三类：

✅ 值得继续：SERP 可切入 + 趋势/数据/社区至少有两个维度的证据。可以开始做产品规划。

⚠️ 可观察：有一定信号但证据不够硬，或者 SERP 竞争不确定。放进观察池，下周再跑一次看变化。

❌ 不值得做：大站锁死、品牌词、娱乐噪音、或者趋势在下降。直接排除。

这轮从几百个种子词最后产出：13 个值得继续，3 个可观察，其余全部淘汰。
